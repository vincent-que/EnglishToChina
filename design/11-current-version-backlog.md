# 当前版本未完成功能与后续开发清单

> 生成日期：2026-05-17  
> 版本范围：当前 Electron + React + Python 本地桌面版本，已包含 Kimi 翻译、PDF/DOCX 基础转换、术语表导入、任务队列、免安装包输出。  
> 用途：后续开发存档、排期、验收对照。

---

## 1. 当前版本概况

当前版本已经具备本机运行和核心文档翻译链路：

- Electron 桌面端可打包为免安装目录：`release/win-unpacked/`
- React UI 已包含工作台、设置、预览、激活页。
- Python Worker 已接入 PDF/DOCX 解析、翻译、重建。
- PDF 采用 `pdf2docx -> WordParser -> 翻译 -> WordRebuilder/PDFRebuilder` 链路。
- Kimi/Moonshot 已作为默认翻译引擎之一接入。
- 翻译进度已能从 Python Worker 传回主进程并更新 UI。
- 批量任务已按 `concurrentLimit` 排队。
- 术语表支持导入 CSV/TSV/TXT/JSON，并注入翻译 Prompt。
- 翻译历史已持久化到本地 JSON。
- 已生成并验证免安装版可启动。

但当前版本仍未达到正式商用发布状态。主要缺口集中在授权商业化、OCR、正式安装包、资源完备性、UI 文案编码、PDF 高保真输出和测试体系。

## 2. 已完成但仍需增强的模块

### 2.1 PDF/DOCX 翻译链路

**当前状态**

- PDF 先转 DOCX，再进行文本抽取和替换。
- Word 输出已优先基于 PDF 转出的中间 DOCX 模板替换文本，减少乱版。
- 表格单元格支持合并单元格去重和按原文兜底匹配。
- Kimi 实测可完成样例发票翻译，剩余英文主要是人名、公司名、Swift Code、EXW 等专名或贸易术语。

**未完成 / 风险**

- PDF 输出在没有 Microsoft Word、LibreOffice 或 `docx2pdf` 时，会走 ReportLab 纯 Python 兜底，版式保真度仍有限。
- 复杂 PDF 的多页页眉页脚、图片、复杂嵌套表格、绝对定位文本仍可能变形。
- 中文文本变长后，部分单元格可能出现换行拥挤或底部贴边。
- 当前没有自动视觉回归测试，无法批量判断“是否乱版”。

**后续建议**

- P0：优先在 Windows 发布环境内置或检测 LibreOffice/Word 转换能力。
- P1：增加 PDF 样例集，对输出 PDF 渲染成图片做页面级对比。
- P1：为表格重建增加字体缩放、行高自适应、最小字号策略。

### 2.2 翻译引擎

**当前状态**

- 支持 DeepSeek、通义千问、GLM、Moonshot/Kimi、百川、硅基流动等 OpenAI-compatible 接口。
- 默认引擎已切到 `kimi`。
- Kimi Prompt 已增强为“只返回简体中文译文”，并清洗 `<text>` 包装标签。
- API 多次失败后会抛错，不再静默返回原英文，避免用户误以为翻译完成。

**未完成 / 风险**

- API Key 已优先使用 Electron `safeStorage` 加密后写入 settings JSON；若系统安全存储不可用，仍会回退为普通本地保存，后续需增加明确提示。
- 设置页已提供“测试连接”按钮，可验证当前引擎和 API Key。
- 没有按模型上下文长度动态分块。
- 没有请求限速、失败暂停、余额不足识别等策略。
- 专名/贸易术语是否翻译缺少用户偏好控制。

**后续建议**

- 已完成：设置页增加“测试连接”。后续 P1：增加 Key 显示/隐藏切换和凭据加密。
- P1：对 API 错误码做结构化识别：限流、余额不足、Key 无效、网络失败。
- P1：增加“保留专名/翻译专名”偏好。

### 2.3 术语表

**当前状态**

- 后端 `TermService` 已支持导入 CSV/TSV/TXT/JSON。
- 设置页可导入并勾选术语表。
- 翻译时会注入实际术语条目。

**未完成 / 风险**

- 没有术语表编辑、删除、导出。
- 没有导入预览和重复项合并策略。
- 没有术语命中统计。
- 没有与具体任务绑定的术语使用记录。
- CSV 解析较简单，复杂引号、逗号转义场景未处理。

**后续建议**

- P1：增加术语表管理页或设置页内编辑弹窗。
- P1：导入时展示解析结果和错误行。
- P2：增加任务结果里的术语命中报告。

### 2.4 批量任务与历史记录

**当前状态**

- 批量任务按 `concurrentLimit` 排队执行。
- 任务历史持久化到本地 JSON。
- 重启后未完成任务会标记为中断。

**未完成 / 风险**

- 没有真正的任务暂停/恢复。
- 取消任务只改状态，无法中断当前 Python Worker 内正在进行的 API 请求。
- 历史记录没有清理按钮、筛选、失败重试。
- 历史记录仍是 JSON 文件，尚未迁移 SQLite。

**后续建议**

- P1：增加失败重试、清空历史、打开输出目录。
- P2：引入 SQLite，持久化任务、术语表、设置快照。
- P2：Python Worker 支持任务级取消令牌。

## 3. 仍未完成的核心功能

### 3.1 授权系统

**当前状态**

- 激活页 UI 已存在。
- IPC 通道 `license:activate`、`license:validate` 已暴露。
- 当前后端已拆出 `LicenseService` 基础壳：
  - `license:activate` 支持授权码格式校验和本地待验证缓存。
  - `license:validate` 从本地授权缓存返回状态。
  - 尚未接入授权服务器和签名令牌校验。

**缺失内容**

- `src/main/services/license.service.ts`
- 授权码激活与签名令牌校验。
- 授权码格式校验。
- 本地授权缓存。
- 离线宽限期。
- 授权服务器 API 对接。
- 黑名单、撤销与异常授权码处理。
- 未授权输出水印或功能限制策略。

**优先级**

P0，正式商用发布前必须完成。

### 3.2 OCR 扫描版 PDF

**当前状态**

- `python/ocr/__init__.py` 存在，但没有 OCR 实现。
- `requirements.txt` 未包含 PaddleOCR 或其它 OCR 依赖。
- 扫描版 PDF、图片型 PDF 暂不支持可靠翻译。

**缺失内容**

- `python/ocr/ocr_engine.py`
- OCR 依赖和模型资源管理。
- PDF 页面图片化检测。
- OCR 文本块坐标映射。
- OCR 后的重建策略。

**优先级**

P2。若产品目标客户大量处理扫描件，可提升到 P1。

### 3.3 正式安装包

**当前状态**

- 免安装版已成功生成：`release/win-unpacked/`
- 已压缩：`release/EnglishToChina-win-unpacked.zip`
- NSIS 安装包打包失败，原因是 electron-builder 下载/解压 `winCodeSign` 时需要创建符号链接，本机权限不足。

**缺失内容**

- 正式 NSIS 安装包。
- 签名证书。
- 安装/卸载流程验证。
- 桌面快捷方式和开始菜单验证。
- 打包后的 Python embedded runtime。

**优先级**

P0。当前可本地运行，但不适合作为正式分发包。

### 3.4 Python Embedded Runtime

**当前状态**

- 打包后的程序会优先查找 `resources/python-embedded/python.exe`。
- 如果不存在，会回退使用系统 `python`。
- 当前免安装包依赖用户机器已经装好 Python 和相关 pip 包。

**缺失内容**

- `python-embedded/` 运行时目录。
- Python 依赖预安装。
- PyMuPDF、pdf2docx、python-docx、reportlab 等依赖的离线打包验证。
- 启动时依赖缺失提示。

**优先级**

P0。否则换一台没有 Python 环境的电脑可能无法翻译。

### 3.5 自动更新

**当前状态**

- IPC `app:checkUpdate` 固定返回 `{ hasUpdate: false }`。

**缺失内容**

- 更新服务器。
- 版本检查。
- 下载更新。
- 差分/全量更新策略。
- 更新失败回滚。

**优先级**

P3。首版可暂缓。

## 4. UI 与用户体验未完成项

### 4.1 文案编码问题

**当前状态**

- 多个 TS/TSX/CSS/JSON 文件中的中文文案在终端或源码中出现乱码。
- 编译可通过，但用户界面可能显示乱码文案，尤其是侧边栏、按钮、状态标签。

**影响**

- 严重影响产品观感。
- 后续维护者难以阅读代码。

**建议**

- P0：统一源码为 UTF-8。
- P0：逐页修复可见中文文案。
- P1：抽离 i18n 文案常量，避免内联散落。

### 4.2 设置页仍不完整

**缺失内容**

- API Key 显示/隐藏切换。
- 测试连接已完成，需补充错误码展示和帮助链接。
- 输出目录选择。
- PDF 输出策略选择：Word/LibreOffice/ReportLab。

P1。

### 4.3 预览页能力弱

### 4.4 激活页只有 UI 壳

**缺失内容**

### 4.5 偏好设置页

## 5. 数据存储与安全

### 5.1 SQLite 未接入

**当前状态**

- 设置、术语表、历史记录均使用 JSON 文件保存。

**风险**

- 并发写入和损坏恢复能力弱。
- 难以查询、筛选、迁移。
- 无法支撑授权、历史、术语、日志等复杂数据。

**建议**

- P1：引入 SQLite。
- P1：设计 schema：settings、translation_tasks、term_tables、term_entries、licenses、activation_logs。

### 5.2 API Key 安全

**当前状态**

- API Key 明文保存在本地设置文件。
- 日志中已做末尾脱敏，但存储未加密。

**建议**

- P1：使用 Windows Credential Manager 或 Electron safeStorage。
- P1：日志禁止打印完整 Key。
- P1：导出诊断日志时自动脱敏。

### 5.3 日志与诊断

**当前状态**

- 主进程和 Python Worker 有 console/logger 输出。
- 没有统一日志文件和诊断导出。

**建议**

- P1：增加日志目录。
- P1：记录任务阶段、错误码、耗时、文件大小、引擎。
- P2：设置页提供“导出诊断日志”。

当前项目缺少自动化测试体系。

### 必补测试

- Python 单元测试：
  - WordParser 表格合并单元格解析。

  - WordRebuilder 表格按原文兜底替换。

  - TranslateService 队列并发控制。
  - TermService 导入 CSV/JSON。
  - SettingsService 读写。

  - 样例 PDF -> DOCX。

  - 断网/API Key 错误时任务失败状态。

  - 设置保存后工作台使用最新引擎和 Key。

## 8. 建议开发优先级

### P0：本地可用与发布前阻塞

1. 修复所有可见中文乱码文案。
2. 打包 Python embedded runtime 和依赖。
3. 补充中文字体资源，确保 PDF 兜底可渲染中文。
4. 解决 NSIS 安装包打包问题。
5. 实现授权系统基础版。
6. 增加 API Key 显示/隐藏切换与安全存储。

### P1：核心体验提升

1. SQLite 数据库替代 JSON 存储。
2. 完整术语表管理：编辑、删除、导出、预览。
3. 翻译错误码体系和用户友好提示。
4. PDF 输出高保真策略：优先 LibreOffice/Word，ReportLab 兜底优化。
5. 任务历史筛选、重试、清理。
6. 日志文件与诊断导出。

### P2：功能完善

1. OCR 扫描版 PDF。
2. 文档预览和原文/译文对照。
3. 单段重新翻译。
4. 偏好设置页。
5. 深色模式完整视觉 QA。

### P3：发布后运维

1. 自动更新。
2. 代码混淆和安全加固。
3. 授权后台。
4. 使用统计和崩溃上报。

## 10. 近期验收建议

下一阶段建议以“能给小范围用户试用”为目标，验收标准如下：

- 在一台干净 Windows 电脑上，无需手动安装 Python 即可完成 PDF/DOCX 翻译。
- 设置页输入 Kimi API Key 后，可测试连接成功。
- 样例发票 PDF 输出 DOCX 后，无大段英文漏翻。
- 样例发票 PDF 输出 PDF 后，表格不重复列、不明显重叠。
- 未授权状态有明确限制或水印。
- 应用文案无乱码。
- NSIS 安装包可安装、卸载、创建桌面快捷方式。
