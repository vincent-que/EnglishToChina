# 客户初版反馈整改路线图

> 日期：2026-05-18  
> 背景：初版 ZIP 发给客户后，客户反馈翻译任务一直停留在“排队中”；同时希望 App 开箱即用，不再自行配置 Kimi Key。

## 1. 客户反馈

- 初版 ZIP 包不能执行翻译，任务界面停留在“排队中”。
- 客户希望打开 App 后直接使用，不要自己配置模型 Key。
- “翻译预览”页面对当前版本偏多余。
- Electron 原生菜单是英文，需要中文化或隐藏。
- 后续重点应转向用户信息、授权码和服务端统一模型调度。

## 2. 产品方向

- 客户端默认走服务端翻译代理，不在客户机器保存模型 Key。
- 授权采用月度离线码：每月月初更新授权码，有效期到下月月初。
- 授权码不绑定设备，不采集设备指纹，不限制设备数量。
- 服务端维护模型池，按成本优先自动切换：GLM/低价模型、DeepSeek/硅基流动、Kimi/Qwen 兜底。

## 3. 已落地整改

- 前端先创建任务再调用主进程，避免后端快速失败时前端漏接错误事件。
- `translation:start` 支持前端传入 taskId，错误会回写到同一条任务。
- 默认翻译模式改为服务端代理；本地 API Key 仅作为备用模式。
- 账户信息页支持月度授权码状态、有效期和剩余天数展示。
- 新增 `npm.cmd run license:generate -- YYYYMM 数量`，用于生成当月授权码。
- 移除侧边栏“翻译预览”入口。
- Windows 客户版隐藏默认英文原生菜单。
- 环境自检增加授权状态和翻译服务连接状态。

## 4. 仍需服务端配套

- `GET /api/health`：客户端自检使用。
- `POST /api/license/validate`：服务端二次校验授权码。
- `POST /api/translate`：接收授权码、文本、风格、术语表，返回译文。
- 服务端日志必须脱敏，不打印完整模型 Key、完整授权码或客户文档全文。

## 5. 验收标准

- 干净 Windows 环境解压 ZIP 后可启动。
- 输入有效月度授权码后，不填写 Kimi Key 也可以翻译样例 DOCX/PDF。
- 授权过期、服务端未配置、服务端不可用、模型全部失败时，任务显示明确中文错误，不停留在“排队中”。
- 侧边栏只保留工作台、翻译设置、账户信息。
- 原生菜单无英文默认项。

## 6. 风险

- 没有服务端地址时，客户版只能显示明确错误，不能真正翻译。
- 月度离线码只适合小范围试用；正式商用仍建议服务端二次校验。
- 服务端中转会接触文本内容，需要在客户说明中明确数据处理方式。
